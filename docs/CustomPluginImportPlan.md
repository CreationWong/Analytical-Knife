# 用户自定义 HTML 插件导入功能计划书

## 1. 目标

本方案仅围绕以下需求展开：

1. 用户在单独的“插件管理”界面导入已经编译好的 HTML
2. 用户在插件管理界面配置新插件的侧边栏位置
3. 用户在插件管理界面选择新插件的侧边栏图标
4. 导入完成后，插件立即注册到侧边栏
5. 程序在根目录生成对应的 XML 注册信息

本次方案不再采用“用户自带插件清单文件”的思路，插件元数据由程序内的插件管理界面维护。

## 2. 现状判断

当前项目是 `React + Tauri`，侧边栏来源于前端静态注册表：

- `src/registry/index.ts` 保存内置工具定义
- `src/App.tsx` 根据注册表生成侧边栏树
- 工具组件通过 `lazy(import(...))` 在构建期绑定

这意味着：

1. 用户导入的 HTML 插件不能走现有编译期组件注册方式
2. 必须新增“运行时插件注册表”
3. HTML 插件需要一个通用宿主页面承载

## 3. 总体方案

采用“内置工具注册表 + 用户 HTML 插件注册表”双轨结构：

1. 内置工具继续保留现有 `TOOLS_REGISTRY`
2. 新增用户插件注册数据文件，例如 `custom-plugins.json`
3. `App.tsx` 启动时合并两类注册数据
4. 用户插件统一由 `HtmlPluginHost` 承载
5. 插件管理页负责：
   - 导入 HTML
   - 编辑插件名称
   - 配置侧边栏路径
   - 配置排序位置
   - 选择图标
   - 删除插件
   - 重新生成 XML

## 4. 用户流程

### 4.1 插件导入流程

用户进入“插件管理”页面后：

1. 点击“导入 HTML 插件”
2. 选择一个已经编译好的 HTML 入口文件，例如 `index.html`
3. 程序自动识别 HTML 所在目录作为插件资源根目录
4. 页面展示插件配置表单
5. 用户填写或调整：
   - 插件名称
   - 插件说明
   - 侧边栏位置
   - 排序序号
   - 图标
   - 显示宽度模式
6. 点击保存
7. 程序复制 HTML 文件及其同目录资源到应用托管目录
8. 插件立即出现在侧边栏
9. 同步生成或刷新 XML 注册信息

### 4.2 插件管理页能力

插件管理页建议放在：

- `设置/插件管理`

页面能力建议包含：

- 导入 HTML
- 编辑插件元数据
- 查看当前所有自定义插件
- 切换启用/禁用
- 删除插件
- 手动重新生成 XML
- 查看 XML 生成结果和失败原因

## 5. 导入对象定义

### 5.1 导入单位

导入入口是“已编译 HTML 文件”，不是源码目录，不是 React 组件，不是用户提供的清单文件。

程序在用户选中 `index.html` 后：

1. 取该文件所在目录作为插件根目录
2. 复制该目录下的静态资源到程序托管目录
3. 将选中的 HTML 记录为插件入口文件

这样可以兼容常见前端构建产物，例如：

```text
dist/
├── index.html
├── assets/
│   ├── index.js
│   └── index.css
└── favicon.ico
```

### 5.2 资源处理规则

建议规则如下：

1. HTML 入口文件必须存在
2. HTML 所在目录作为完整资源目录复制
3. 插件运行时保留原始相对路径关系
4. 不支持跨目录引用插件根目录之外的资源

这样可以保证 HTML 中的：

- `./assets/index.js`
- `./assets/index.css`
- `./favicon.ico`

在导入后仍然有效。

## 6. 插件管理页表单设计

### 6.1 必填项

建议至少提供以下字段：

- `插件名称`
- `HTML 入口文件`
- `侧边栏目录路径`
- `图标`

### 6.2 建议字段

建议增加以下字段，便于侧边栏控制：

- `插件说明`
- `排序序号`
- `是否启用`
- `最大宽度模式`

### 6.3 侧边栏位置配置

“侧边栏位置”不建议只做成一个自由字符串，应拆成两个概念：

1. `挂载路径`
   - 例：`用户插件/常用工具`
   - 表示插件挂在哪个目录树下

2. `排序序号`
   - 例：`100`
   - 数值越小越靠前

最终显示名称直接使用插件名称，不再要求用户把插件名称拼进路径里。

推荐最终渲染逻辑：

- 完整路径 = `挂载路径 + 插件名称`
- 同级节点按 `排序序号 -> 名称` 排序

### 6.4 图标配置

图标建议由插件管理页下拉选择，不允许用户直接传 React 组件。

V1 推荐方案：

- 维护一组内置图标键值
- 用户从可视化列表中选择
- 系统保存 `iconKey`
- 渲染时将 `iconKey` 解析成对应图标组件

这样能和现有侧边栏系统兼容，也比用户上传任意图标文件稳定。

## 7. 数据结构设计

### 7.1 用户插件注册数据

建议新增持久化文件：

- `appDataDir()/custom-plugins.json`

单条记录建议包含：

```json
{
  "id": "plugin_20260531_ab12cd",
  "name": "HTML 示例工具",
  "description": "用户导入的 HTML 工具",
  "enabled": true,
  "iconKey": "IconCode",
  "sidebarGroupPath": "用户插件/常用工具",
  "sidebarOrder": 100,
  "entryFile": "index.html",
  "pluginRoot": ".../plugins/plugin_20260531_ab12cd",
  "windowMaxWidth": "none",
  "createdAt": "2026-05-31T12:00:00Z",
  "updatedAt": "2026-05-31T12:00:00Z"
}
```

### 7.2 运行时侧边栏模型

建议新增一套统一运行时定义：

```ts
type RuntimeToolDefinition = BuiltinToolDefinition | CustomHtmlPluginDefinition;
```

用户插件需要的最小字段：

- `source: 'custom-plugin'`
- `id`
- `name`
- `description`
- `path`
- `iconKey`
- `sidebarOrder`
- `entryFile`
- `pluginRoot`
- `windowMaxWidth`

## 8. 文件托管方案

### 8.1 托管目录

导入后的 HTML 插件不应继续依赖原始位置，建议统一复制到：

- `appDataDir()/plugins/<pluginId>/`

原因：

1. 用户删除原始 `dist` 目录后不影响插件运行
2. 后续删除、重命名、禁用插件更容易管理
3. XML 生成时可以稳定引用托管后的路径

### 8.2 导入时处理规则

导入时：

1. 生成插件 ID
2. 新建托管目录
3. 复制选中 HTML 所在目录的全部内容
4. 写入 `custom-plugins.json`

删除时：

1. 删除注册记录
2. 删除托管目录
3. 刷新 XML

## 9. HTML 插件承载方案

### 9.1 宿主组件

建议新增：

- `src/components/HtmlPluginHost.tsx`

作用：

- 接收插件入口文件路径
- 将本地路径转换成可加载 URL
- 在主区域中显示 HTML 插件

### 9.2 加载方式

推荐使用 Tauri 的 `convertFileSrc()` 将入口 HTML 转成本地资源 URL，再交给 `iframe` 加载。

理由：

1. HTML、CSS、JS 与主应用隔离
2. 相对路径资源天然可用
3. 不需要把 HTML 内容注入 React DOM
4. 后续需要通信时可扩展 `postMessage`

### 9.3 安全策略

HTML 插件应当运行在受控容器里，不应直接获得主应用上下文。

建议：

- 使用 `iframe`
- 设置 `sandbox`
- 默认只开放必需能力

推荐最小能力：

- `allow-scripts`
- `allow-same-origin`
- `allow-forms`

默认不开放：

- 顶层跳转
- 任意弹窗
- 对主窗口对象的直接控制

## 10. XML 生成方案

### 10.1 输出目标

生成文件：

- `plugins.xml`

### 10.2 生成来源

XML 不依赖用户手写配置文件，而是直接从 `custom-plugins.json` 和当前有效注册状态生成。

这样做的好处：

1. 页面配置和 XML 输出只有一个真实数据源
2. 用户修改侧边栏位置或图标后，XML 可直接重建
3. 不会出现“页面显示一套、XML 另一套”的分叉

### 10.3 XML 结构建议

建议输出结构：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<pluginRegistry generatedAt="2026-05-31T12:00:00Z">
  <plugin
    id="plugin_20260531_ab12cd"
    name="HTML 示例工具"
    enabled="true"
    type="html">
    <sidebar groupPath="用户插件/常用工具" order="100" icon="IconCode" />
    <entry file="index.html" />
  </plugin>
</pluginRegistry>
```

### 10.4 生成时机

建议在以下时机自动生成：

1. 成功导入插件后
2. 修改插件元数据后
3. 删除插件后
4. 用户点击“重新生成 XML”后

### 10.5 生成位置

需求要求写入程序根目录，因此建议：

1. 开发态写入仓库根目录
2. 发布态优先写入可执行文件同级目录

但这里必须明确一个现实约束：

- Windows 安装目录可能不可写

所以实现上必须支持：

1. 先尝试写入根目录
2. 失败时返回明确错误
3. 插件管理页提示用户失败原因

如果后续要强制保证成功，只能配合管理员权限或调整安装位置。

## 12. 侧边栏集成方案

### 12.1 注册表合并

`App.tsx` 不再只读取静态 `TOOLS_REGISTRY`，而是：

1. 加载内置工具
2. 加载用户插件
3. 合并成统一运行时列表
4. 构造树形侧边栏

### 12.2 路径处理

建议用户插件不要直接使用旧的 `path` 模式拼完整工具路径，而是运行时生成：

- `path = sidebarGroupPath + '/' + name`

这样改名时不需要用户重复维护完整路径。

### 12.3 排序处理

现有树结构基本按注册顺序显示，无法精确控制用户插件位置。为了满足“插件页面控制位置”，建议新增排序逻辑：

1. 同级目录节点支持排序字段
2. 叶子工具支持 `sidebarOrder`
3. 排序规则为：
   - 先按 `sidebarOrder`
   - 再按 `name`

## 13. 图标方案

V1 推荐只支持系统预置图标，不支持用户上传任意 SVG/PNG 作为侧边栏图标。

原因：

1. 当前侧边栏依赖 React 图标组件
2. 预置图标更容易统一风格
3. 运行时无需处理图像缩放、主题适配和异常资源

具体实现：

1. 建立 `iconKey -> IconComponent` 字典
2. 插件管理页展示可选图标列表
3. 保存时只记录 `iconKey`
4. 渲染时回退默认图标

## 16. 风险与处理

### 风险 1：用户只选了 HTML，资源不完整

处理：

- 以 HTML 所在目录为整体复制单位
- 导入前提示“请确保已选择编译产物入口文件”

### 风险 2：安装目录不可写，XML 生成失败

处理：

- Rust 返回明确错误
- 插件管理页展示失败原因
- 后续再决定是否增加降级导出

### 风险 3：用户配置的侧边栏路径冲突

处理：

- 允许共用目录
- 工具叶子节点使用唯一插件 ID 管理
- 展示层按名称和排序序号区分

### 风险 4：动态插件破坏主界面

处理：

- 强制 iframe 隔离
- 不把 HTML 注入 React DOM
- 默认不开放主应用上下文

## 17. 最小可交付版本

建议首版范围控制为：

1. 只支持导入已编译 HTML 入口文件
2. 自动复制 HTML 所在目录
3. 插件管理页可配置：
   - 名称
   - 说明
   - 挂载路径
   - 排序序号
   - 图标
4. 侧边栏可立即显示插件
5. 自动生成 `plugins.xml`

这一版已经能满足你当前提出的核心需求，而且改动面相对可控。

## 18. 结论

重写后的方案核心变化是：

1. 不要求用户提供插件清单文件
2. 以“导入已编译 HTML”作为唯一入口
3. 插件名称、侧边栏位置、图标全部由插件管理页控制
4. 程序内部维护唯一注册数据源，再生成 XML

这条路线和当前仓库更兼容，也更符合“用户在一个独立插件页面完成导入和配置”的使用方式。
